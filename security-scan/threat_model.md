## Overview

TikTok LIVE Companion 0.5.0 is a standalone Chrome Manifest V3 browser extension for `https://www.tiktok.com/*`. Its primary runtime surfaces are:

- `background.js`: privileged service worker using `chrome.storage`, `chrome.tabs`, `chrome.scripting`, `chrome.sidePanel`, and passive `chrome.webRequest` observation.
- `content-core.js` and `content.js`: isolated-world content logic that scans TikTok DOM, embedded metadata, performance entries, public profile pages, chat elements, media elements, captions, quality controls, and player controls.
- `proto-main.js` and `hook.js`: optional MAIN-world WebSocket wrapper and protobuf/gzip decoder registered at `document_start` to observe TikTok LIVE messages.
- `sidepanel.html`, `sidepanel.js`, and `sidepanel.css`: extension-owned UI for displaying chat, captions, profile and LIVE statistics, signed FLV/HLS media URLs, diagnostics, speech synthesis, and operator-triggered TikTok player actions.
- `manifest.json`: permission and origin policy for the extension.
- `tiktok-live-companion-extension-0.5.0.zip` and `tiktok-live-companion-plugin-0.5.0.zip`: release artifacts. They are supply-chain and distribution surfaces rather than active browser runtime components; the plugin archive also packages the browser extension with supporting documentation, tests, and tooling.

The extension has no application backend, remote analytics, form submission service, native host, authentication system, or declared externally connectable interface. It relies on the browser’s extension isolation and TikTok’s authenticated page session but does not request the `cookies` permission. A public-profile fallback request in `content.js` explicitly uses `credentials: "omit"`.

Security-relevant assets include:

- Integrity of the installed extension package, its locally bundled code, and its declared permissions.
- Confidentiality of TikTok page content observed in an authenticated tab, including chat, captions, profile metadata, LIVE statistics, AI-summary text, WebSocket-derived events, and page URLs.
- Confidentiality of complete FLV/HLS URLs. These may contain time-limited signed query parameters and therefore should be treated as bearer-like media access material until expiry.
- Integrity of user-directed player actions, quality changes, caption activation, hook registration, tab reloads, clipboard writes, speech output, and local exports.
- Availability and responsiveness of TikTok pages, the MAIN-world decoder, the service worker, and the side panel when processing remote DOM, metadata, or WebSocket input.
- Isolation of state among browser tabs. `background.js` stores runtime data under per-tab keys and removes tab state when the tab closes.

The key security invariants are that TikTok-controlled data must remain data rather than executable extension content; only intended extension contexts and explicit user actions should reach privileged operations; state and commands must remain bound to the correct tab; media URLs and captured content must not leave the browser except through clear user-directed copy or export operations; MAIN-world interception must not broaden TikTok page privileges; and hostile or malformed remote input must not exhaust browser resources or destabilize unrelated tabs.

## Threat Model, Trust Boundaries, and Assumptions

- **User and extension UI boundary:** The user is the trusted operator. The side panel is extension-owned and may request privileged actions through `chrome.runtime.sendMessage`. Copying a media URL, exporting captions or diagnostics, enabling persistent hook registration, speech synthesis, and manipulating the TikTok player are intentional user-facing capabilities. The model assumes these actions are initiated by the user through the genuine extension UI.
- **Side panel and service-worker boundary:** `sidepanel.js` selects the active tab, includes its numeric `tabId` in runtime messages, and disables page actions unless the active URL begins with `https://www.tiktok.com/`. `background.js` is the central privileged dispatcher and must preserve authorization, action allowlisting, input validation, and correct-tab binding for every message. No `externally_connectable` declaration exists, so ordinary web pages are not expected to invoke its runtime API directly.
- **Service worker and content-script boundary:** The service worker forwards a fixed set of commands to the content script, including scanning, caption activation, quality changes, player actions, and debug configuration. Content-derived state returns across the same channel. The content script executes in Chrome’s isolated world and does not directly expose privileged Chrome APIs to page JavaScript.
- **Content script and TikTok DOM boundary:** All TikTok DOM nodes, script text, JSON metadata, resource timing entries, profile responses, labels, captions, chat messages, media attributes, and player elements are attacker-controlled inputs. This remains true even when the normal producer is TikTok, because creators, chat participants, network intermediaries on non-TLS CDN requests, compromised platform content, browser instrumentation, or future page changes may influence those values.
- **Isolated world and MAIN world boundary:** `proto-main.js` and `hook.js` run in the TikTok page’s MAIN world and therefore share a realm with untrusted TikTok JavaScript. The hook wraps `window.WebSocket`, observes binary frames, and sends decoded data through `window.postMessage`. `content.js` accepts only messages whose source is the same window, whose origin equals `location.origin`, and whose envelope has the expected source marker and version. Those checks provide routing and origin constraints, but the marker is not a secret; the model therefore treats all page-to-content `postMessage` payloads as untrusted and requires schema, size, type, and semantic validation downstream.
- **Browser network boundary:** `content.js` fetches only an encoded TikTok profile handle over HTTPS with credentials omitted. `background.js` passively observes requests to configured TikTok CDN families. `content-core.js` accepts media URLs only when their scheme is HTTP(S), their hostname exactly matches or is a subdomain of an allowlisted CDN suffix, and their path/query indicates FLV, HLS, or audio. The manifest grants `*://` access to these CDN families, so both TLS and non-TLS CDN traffic are within the declared observation surface.
- **Browser storage boundary:** Captured tab state is kept in `chrome.storage.session`, including full media URLs, captions, chat, profile information, summaries, and debug entries. Lists are bounded (`MAX_MEDIA`, `MAX_CAPTIONS`, `MAX_CHAT`, `MAX_EVENT_IDS`, and `MAX_DEBUG`), and tab state is removed on tab closure. Only user preferences such as hook autostart and speech settings persist in `chrome.storage.local`.
- **Clipboard, download, and speech boundaries:** Full media URLs are copied only after a button click. Caption JSONL and debug JSON are downloaded only after explicit user actions. The debug report replaces every media URL query value with `REDACTED`; caption exports contain captured caption records. Chat speech is opt-in, uses browser-local speech synthesis, caps its pending queue, and is stopped on tab changes unless the user enables persistence.
- **Installation and update boundary:** The two ZIP files may be distributed independently of the source directory. The model assumes users obtain an authentic archive and that Chrome’s extension installation/update mechanism and the host operating system are not compromised. Package substitution, unintended permission expansion, or modification of bundled scripts is nevertheless in scope as a repository supply-chain threat.
- **Browser platform assumptions:** Chrome 114 or later correctly enforces Manifest V3 extension isolation, extension-page content security policy, host permissions, runtime-message provenance, and process sandboxing. Other installed extensions, developer tools, the browser binary, and the operating system are assumed non-malicious.
- **Product-scope assumptions:** The extension is intended to observe and locally present data already available to the active TikTok page, not to bypass TikTok authentication or authorization. It does not submit reports, send chat messages, change account settings, or upload captured data. Opening TikTok’s report dialog and operating player controls are in scope; compromise of TikTok’s own servers or policy disputes over intentional media-link copying are not extension vulnerabilities unless repository code creates an additional confidentiality, integrity, or authorization failure.
- **Availability assumptions:** TikTok markup, protobuf layouts, and player controls may change without notice. Compatibility failures caused solely by legitimate upstream changes are functional defects, while attacker-induced resource exhaustion, unsafe parsing, privilege crossing, or unintended actions remain security-relevant.

## Attack Surface, Mitigations, and Attacker Stories

- **Manifest permissions and origin reach:** `manifest.json` grants `activeTab`, `scripting`, `sidePanel`, `storage`, `tabs`, and `webRequest`, with hosts restricted to `www.tiktok.com` and enumerated CDN suffixes. There are no wildcard all-site permissions, cookie access, native messaging, remote-code declarations, or web-accessible resources. A relevant attacker story is a malicious or compromised TikTok page attempting to turn these permissions into broader browser or account access. Privileged behavior must remain limited to the manifest’s origins and the fixed action set.
- **Runtime message dispatch:** Messages drive storage reads and writes, hook registration, tab reloads, content-script commands, diagnostics, and player actions. The absence of an externally connectable interface limits ordinary web-origin callers, while per-tab state keys limit accidental cross-tab mixing. Realistic threats include a compromised extension context, confused active-tab state, malformed content-script payload, or spoofed page-derived event causing a command or state update to be applied to the wrong tab. Message type allowlisting, sender/tab association, numeric tab validation, TikTok URL checks for hook/reload operations, and bounded values are the relevant invariants.
- **MAIN-world WebSocket interception:** `hook.js` replaces `window.WebSocket` with a proxy while retaining the native constructor and attaches passive listeners. It strips WebSocket query strings from the stored endpoint and does not modify outbound or inbound frames. `proto-main.js` exports a frozen decoder API under a symbol, and the hook marks installation with a non-configurable symbol. Realistic attacker stories include page code spoofing hook messages, tampering with the shared MAIN-world environment before registration, supplying malformed protobuf, or sending compressed data that consumes excessive CPU or memory. Existing robustness controls include a 16 MiB pre-decode frame limit, strict protobuf wire-type and length checks, error containment, and omission of common parse errors from status reporting.
- **Page-to-extension event bridge:** `window.postMessage` carries captions, chat, LIVE events, and hook status into the isolated content script. Source, same-origin, source-marker, and version checks reject unrelated frames and windows, but same-page scripts can reproduce the marker. Payloads therefore must never be interpreted as authority. Attacker stories include forged viewer statistics, injected chat or caption records, oversized nested objects, misleading endpoint errors, and attempts to influence later UI or speech behavior.
- **Metadata, JSON, DOM, and resource scanning:** `content.js` scans all page scripts, selected DOM regions, resource timing entries, profile-page JSON, and frequently changing chat nodes. `content-core.js` recursively visits input with a node budget, tracks visited objects, restricts nested JSON parsing by length and content hints, and rejects non-CDN/non-media URLs. Profile fetch parsing skips scripts over 8 MiB. Realistic threats include deeply nested JSON, huge script collections, pathological strings or object graphs, DOM churn that repeatedly triggers scans, deceptive profile data, and media URLs crafted to exploit parsing or display code.
- **Binary and compressed parsing:** WebSocket frames and gzip payloads are fully attacker-controlled. The protobuf reader rejects invalid field numbers, unsupported wire types, unsafe lengths, and truncated fields, and decoder errors are caught by the hook. Important vulnerability classes are integer/length confusion, decompression amplification, parser hangs, memory exhaustion, and unsafe assumptions about field types. Parser output must remain bounded before it is posted, stored, rendered, or spoken.
- **DOM automation and player control:** Caption activation, quality selection, playback, mute, volume, limiter, picture-in-picture, fullscreen, replay, and opening the report dialog operate on TikTok DOM selected by labels and attributes. Most actions originate from explicit side-panel clicks, and player operations are restricted to a fixed action allowlist. Attacker stories include TikTok-controlled markup presenting lookalike elements that match selectors, causing an unintended click, or rapid navigation changing the active target during an operation. Security impact depends on whether the action remains a reversible player operation or crosses into account, publication, purchase, messaging, or report-submission behavior.
- **Audio and speech:** Chat content can originate from arbitrary TikTok participants. `sanitizeChatText` normalizes Unicode, removes emoji sequences, collapses whitespace, and state and speech queues are bounded. The side panel requires opt-in before invoking local `SpeechSynthesisUtterance`. Attacker stories include abusive or deceptive spoken content, excessive message rate, very long utterances, language manipulation, and audio denial of service. Spoken text must not be treated as a trusted instruction, and speech must remain locally controlled and readily stoppable.
- **Rendering and extension-page script execution:** The side panel constructs nodes with `document.createElement` and places remote values through `textContent`; searches use `CSS.escape`. No `innerHTML`, `eval`, dynamic function construction, or remote script loading is present. These controls materially reduce extension-context XSS risk. Attacker stories still include malformed values triggering DOM selector, formatting, accessibility, or rendering failures, and any future change from text sinks to HTML or URL execution sinks would materially change the risk.
- **Media-link confidentiality:** Media discovery combines passive `webRequest` observation, performance entries, and metadata extraction. Hostname suffix checks and media-type classification prevent arbitrary origins from entering the media list. Full accepted URLs are retained in session state, shown in the side panel, and copied only on user action. Realistic attacker stories include disclosure of signed query parameters through shoulder surfing, clipboard access by another local application, unredacted logs, cross-tab confusion, or unintended export. Debug exports specifically redact all media URL query values, but intentional copy exposes the complete URL by design.
- **Captured-content confidentiality:** Chat, captions, profile data, summaries, and statistics may be personal or sensitive even when visible on the page. They remain local in session storage and bounded in count. Caption export and persistent speech behavior require user choices. Attacker stories include another extension context gaining access, the wrong active tab being rendered, a user accidentally sharing an export, or sensitive content appearing in diagnostic entries. The lack of backend transmission and the absence of cookies or analytics reduce remote exfiltration exposure.
- **Diagnostics and error handling:** Debugging is opt-in per tab. Debug reports include page, profile, summary, hook, statistics, player state, redacted media URLs, counts, and debug entries, but not stored chat or caption bodies as top-level report collections. Hook endpoints omit query strings, error strings are truncated, and recorded collections are capped. Threats include secrets appearing in unexpected error text or nested debug detail, misleading forged diagnostics, and a user sharing a report without reviewing page/profile metadata.
- **Storage and tab lifecycle:** State is namespaced by tab ID, rehydrated with defaults, capped, and cleared when a tab closes. Profile data is cached by normalized handle for the browser session and merged into matching tab states. Threats include stale tab identity, handle collision or spoofing, incorrectly merged profiles, race conditions among concurrent messages, and service-worker restarts producing inconsistent state. These usually affect integrity and privacy within one browser profile unless chained to a more privileged action.
- **Release artifacts and local code integrity:** All active JavaScript and CSS is bundled locally, which avoids runtime dependency or CDN-script compromise. The ZIPs are independent distribution objects whose contents and hashes must correspond to reviewed source. Attacker stories include archive replacement, stale or mismatched packaged files, a plugin archive containing additional privileged instructions or tooling, and later manifest changes that silently expand permissions. Reproducible packaging, source-to-archive comparison, published digests, and review of permission changes are the relevant mitigations.
- **Out-of-scope or reduced-relevance stories:** A malicious operating system, compromised Chrome binary, hostile developer mode, another malicious installed extension with sufficient permissions, or a user intentionally installing a modified archive can defeat repository controls and is outside the ordinary runtime threat model. CSRF, server-side request forgery, SQL injection, tenant isolation, password storage, server authorization, and cloud-secret management are not primary classes because the repository has no server, database, account system, or secret-bearing backend. They become relevant only if such services are added.

## Severity Calibration (Critical, High, Medium, Low)

- **Critical:** Reserved for repository behavior that enables reliable arbitrary code execution outside normal extension isolation, browser/OS compromise, or a release/update supply-chain takeover that executes attacker code across a large installed population with the extension’s TikTok access. Systematic, remote, low-interaction theft of authenticated TikTok sessions or equivalent account-control secrets across users could also qualify. A tab-local parser crash, spoofed statistic, or individual signed media URL does not reach Critical without a substantially broader compromise.
- **High:** Applies to reliable extension-context code execution from TikTok-controlled data; unauthorized privileged actions with serious account or cross-origin consequences; persistent compromise of the MAIN-world hook or runtime dispatcher that exposes authenticated page data across TikTok tabs; silent remote exfiltration of chat, captions, profiles, or usable signed media URLs; or a package-integrity failure affecting distributed artifacts. Impact should be substantial and normally require little or no additional user action. The lack of cookie permission, backend transmission, native messaging, and all-site host access limits many plausible issues from escalating beyond this level.
- **Medium:** Applies to meaningful single-user or tab-scoped confidentiality, integrity, or availability failures: a forged page event causing misleading stored or spoken content; leakage of a usable signed stream URL under realistic conditions; cross-tab state confusion; an unintended but reversible player or TikTok UI action; sustained CPU or memory exhaustion from DOM, protobuf, or gzip input; or sensitive metadata included in a user-generated diagnostic export beyond reasonable expectations. Issues that require the victim to enable the hook, visit attacker-influenced TikTok content, or explicitly export/copy data may still be Medium when the resulting harm is material.
- **Low:** Applies to limited spoofing, stale statistics, minor diagnostic metadata exposure, short-lived UI or speech disruption, graceful parser failure, inaccurate labels, or other effects confined to extension presentation with no durable account, confidentiality, or availability consequence. Problems requiring malicious local developer tools, another already-privileged extension, deliberate installation of a modified package, or intentional sharing of a clearly identified export are generally Low or out of scope unless they reveal an independent trust-boundary failure.
